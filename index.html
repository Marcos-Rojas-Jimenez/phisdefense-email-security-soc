Contexto del proyecto

Este proyecto nacio como TFM y fue adaptado como pieza de portfolio profesional. El objetivo fue construir un laboratorio defensivo capaz de demostrar, con evidencias, que los controles de seguridad de correo estaban configurados y validados correctamente.

Problema

Muchas configuraciones de correo incluyen SPF, DKIM o DMARC, pero no siempre existen pruebas practicas ni evidencias claras de que esos controles funcionen ante abuso SMTP, spoofing o errores de autenticacion.

Objetivo

Convertir controles tecnicos de email security en eventos medibles, documentados y visibles desde una perspectiva SOC, usando logs reales y automatizacion.

Resultado

Un dashboard SOC con metricas, categorias, tabla de eventos, evidencias y dataset publico sanitizado, preparado para ser mostrado en entrevistas.

Solucion implementada

La solucion combina servicios reales de correo, procesamiento de logs y visualizacion SOC. La idea central fue no registrar pruebas sin evidencia tecnica.

Arquitectura funcional

  • Postfix como servicio SMTP.
  • OpenDMARC para validacion y rechazo DMARC.
  • OpenDKIM para validacion y gestion DKIM.
  • Dovecot para validaciones IMAPS TLS.
  • Syslog como fuente principal de evidencias.
  • Scripts en Python/Bash para procesar eventos.
  • Dashboard en Python/Dash para KPIs y analisis.

Controles validados

  • DMARC progresivo hasta p=reject.
  • SPF y escenarios SPF fail.
  • DKIM correcto, DKIM roto y DKIM ausente.
  • Rotacion DKIM mediante selector nuevo.
  • STARTTLS SMTP e IMAPS TLS.
  • Open relay test y rechazo de abuso SMTP.
  • Spoofing externo rechazado por DMARC.
Criterio clave: una prueba solo se considero valida si existia evidencia tecnica como respuesta SMTP, transcript TLS, registro syslog, consulta DNS, validacion DKIM/DMARC o milter-reject.

Arquitectura del laboratorio

PhisDefense conecta servicios reales de correo con procesamiento SOC y visualizacion mediante dashboard.

  • Postfix procesa sesiones SMTP.
  • OpenDMARC valida DMARC y aplica rechazos.
  • OpenDKIM participa en validaciones DKIM.
  • Dovecot permite pruebas IMAPS TLS.
  • Syslog conserva evidencias reales.
  • Scripts SOC generan el dataset.
  • El dashboard Python/Dash visualiza KPIs, tablas y categorias.
Metodologia de validacion

Una prueba solo se incorporo al dataset SOC si existia evidencia tecnica verificable.

  • Respuestas SMTP reales.
  • Registros syslog.
  • Validaciones TLS.
  • Consultas DNS.
  • Eventos de Postfix, OpenDMARC y OpenDKIM.
  • Rechazos milter-reject.
Pruebas SOC realizadas
  • auth_fail: fallos SMTP AUTH.
  • recipient_unknown: destinatarios inexistentes.
  • open_relay_test: rechazo de relay abierto.
  • tls_starttls_smtp: pruebas STARTTLS SMTP.
  • tls_imaps: validaciones TLS sobre IMAPS.
  • spf_fail: origen no autorizado.
  • dkim_broken: firma DKIM rota.
  • dkim_absent: ausencia de DKIM.
  • spoofing_reject: intentos externos rechazados por DMARC.
Spoofing y DMARC p=reject

Se valido una politica DMARC progresiva hasta p=reject. Las evidencias incluyeron:

milter-reject: END-OF-MESSAGE
5.7.1 rejected by DMARC policy for phisdefense.com

Los eventos se clasificaron como spoofing_reject.
