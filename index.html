1. Problema que resuelve

PhisDefense nace para resolver un problema muy concreto: no basta con configurar seguridad de correo, tambien hay que probarla, evidenciarla y monitorizarla. El objetivo fue validar de forma practica controles como SPF, DKIM, DMARC, DNSSEC, MTA-STS, TLS-RPT, OpenDKIM, OpenDMARC, OpenARC, Postfix, Dovecot y TLS, y convertir esas pruebas en evidencias SOC.

Problema tecnico

Muchos dominios publican SPF, DKIM o DMARC, pero no siempre se valida si esos controles funcionan ante abuso SMTP, spoofing, errores DKIM, fallos SPF, problemas TLS o autenticaciones fallidas.

Necesidad defensiva

En un entorno Blue Team/SOC, la configuracion debe transformarse en eventos, logs, evidencias y metricas que permitan analizar el comportamiento real del sistema.

Objetivo del proyecto

Desplegar una infraestructura real de correo defensivo, ejecutar pruebas controladas, registrar evidencias y visualizar los resultados en un dashboard SOC.

2. Solucion implementada

La solucion combina configuracion de servidor, seguridad DNS, autenticacion de correo, milters defensivos, transporte seguro, pruebas controladas, parsing de logs y un dashboard en Python/Dash.

1. Capa DNS y dominio: SPF, DKIM, DMARC, DNSSEC, MTA-STS y TLS-RPT como base de autenticacion, confianza y transporte seguro.
2. Servidor de correo: Postfix como MTA, Dovecot para IMAPS TLS y syslog como fuente principal de evidencias.
3. Milters integrados: OpenDKIM, OpenDMARC y OpenARC integrados con Postfix para firmar, validar, rechazar y registrar resultados.
4. Pruebas defensivas: Open relay, SMTP AUTH fail, recipient unknown, SPF fail, DKIM roto, DKIM ausente, rotacion DKIM, lookalike domains y spoofing externo rechazado.
5. Capa SOC: Scripts de procesamiento, dataset SOC, metricas y dashboard con visualizacion de evidencias.

3. Configuracion defensiva del servidor

Esta seccion resume la configuracion real aplicada sobre el servidor y el dominio. El objetivo fue construir una infraestructura defensiva de correo electronico que no solo estuviera configurada, sino tambien probada, evidenciada y monitorizada desde una perspectiva SOC.

1. DNS y confianza del dominio

La capa DNS fue la base de confianza del proyecto. Los controles modernos de email security dependen de registros DNS correctamente publicados y verificables.

  • DNSSEC: refuerzo de autenticidad e integridad de respuestas DNS.
  • SPF: declaracion de servidores autorizados para enviar correo.
  • DKIM: publicacion de clave publica asociada al selector DKIM.
  • DMARC: politica del dominio hasta llegar a p=reject.
  • MTA-STS y TLS-RPT: politicas relacionadas con transporte seguro y reporting TLS.

2. Autenticacion de correo

Se validaron SPF, DKIM y DMARC no solo como registros publicados, sino como controles defensivos sometidos a pruebas positivas y negativas.

  • SPF fail: validacion de origenes no autorizados.
  • DKIM correcto: firma valida con selector publicado.
  • DKIM broken: deteccion de firma invalida o alterada.
  • DKIM absent: diferenciacion de mensajes sin firma.
  • Rotacion DKIM: cambio controlado de selector hacia s2026.
  • DMARC p=reject: rechazo de mensajes no alineados o no autenticados.

3. Postfix y milters defensivos

Postfix fue el nucleo del flujo SMTP. Sobre Postfix se integraron milters defensivos para firmar, validar, rechazar y registrar resultados de autenticacion.

  • Postfix: servidor SMTP principal del laboratorio.
  • OpenDKIM: firma y validacion DKIM.
  • OpenDMARC: evaluacion DMARC y rechazo por politica.
  • OpenARC: soporte ARC para escenarios con intermediarios o forwarding.
  • Syslog: fuente principal para conservar evidencias tecnicas.

4. Transporte seguro

El proyecto tambien valido la seguridad del transporte, separando las pruebas de autenticacion de correo de las pruebas de cifrado y negociacion TLS.

  • STARTTLS SMTP: validacion de negociacion TLS en SMTP.
  • IMAPS TLS: validacion de canal seguro sobre Dovecot.
  • MTA-STS enforce: politica publicada para exigir TLS valido a emisores compatibles.
  • TLS-RPT: reporting documentado para problemas de entrega TLS.

5. Pruebas defensivas controladas

La configuracion se valido mediante pruebas controladas orientadas a detectar abuso, errores de autenticacion, suplantacion y configuraciones inseguras.

  • Open relay test: comprobacion de rechazo de relay no autorizado.
  • SMTP AUTH fail: registro de autenticaciones fallidas.
  • Recipient unknown: rechazo de destinatarios inexistentes.
  • Lookalike domains: analisis de dominios visualmente similares.
  • External spoofing: intentos externos rechazados por DMARC.

6. Evidencias SOC y dashboard

Los eventos generados por el servidor se procesaron y transformaron en datos SOC estructurados. Esta capa conecta la configuracion defensiva con el dashboard.

  • Syslog: origen de eventos reales.
  • Parser SOC: scripts propios para clasificar eventos.
  • CSV SOC: dataset con 190 eventos procesados.
  • Python/Dash: dashboard con metricas, categorias, tablas y evidencias.

4. Automatizacion y dashboard SOC

Tras configurar el servidor, se desarrollo una capa de automatizacion para convertir logs y pruebas tecnicas en informacion SOC visual y reutilizable.

Procesamiento de logs

Scripts propios procesan syslog y extraen eventos relacionados con SMTP, DMARC, DKIM, TLS, autenticacion y rechazos.

Dataset SOC

Los eventos se normalizan en CSV con identificador de prueba, tipo, clasificacion, resultado esperado, ubicacion final y observaciones sanitizadas.

Visualizacion

El dashboard en Python/Dash muestra KPIs, tablas, categorias de eventos, evidencias y vision general del estado defensivo.

Resumen tecnico del proyecto

PhisDefense SOC & Email Security Lab es un laboratorio defensivo de seguridad de correo desarrollado sobre una infraestructura real. El proyecto no se limita a mostrar un dashboard: incluye configuracion DNS, autenticacion de correo, milters integrados con Postfix, transporte seguro, pruebas controladas, procesamiento de logs y visualizacion SOC.

El objetivo principal fue demostrar que los controles defensivos estaban correctamente configurados y que existian evidencias tecnicas de su funcionamiento. Para ello se validaron escenarios reales y negativos relacionados con SPF, DKIM, DMARC, DNSSEC, MTA-STS, TLS-RPT, OpenDKIM, OpenDMARC, OpenARC, STARTTLS SMTP, IMAPS TLS y abuso SMTP.

  • Infraestructura base: servidor Linux con Postfix, Dovecot y syslog.
  • DNS y dominio: DNSSEC, SPF, DKIM, DMARC, MTA-STS y TLS-RPT.
  • Autenticacion de correo: SPF, DKIM y DMARC con politica final p=reject.
  • Milters defensivos: OpenDKIM, OpenDMARC y OpenARC integrados con Postfix.
  • Transporte seguro: pruebas STARTTLS SMTP e IMAPS TLS.
  • Pruebas SOC: auth_fail, recipient_unknown, open_relay_test, SPF fail, DKIM broken, DKIM absent y spoofing_reject.
  • Automatizacion: procesamiento de syslog mediante scripts propios.
  • Dashboard: visualizacion en Python/Dash con metricas, categorias, evidencias y tabla SOC.

Como resultado, el dashboard agrega 270 pruebas totales y 190 eventos SOC procesados. Esta pagina conserva una version publica, explicativa y sanitizada del proyecto para portfolio.

DNS, DNSSEC y registros del dominio

La configuracion DNS fue una parte fundamental del proyecto, porque muchos controles modernos de seguridad de correo dependen directamente de registros DNS publicados correctamente. En PhisDefense se trabajaron registros relacionados con autenticacion de correo, politicas de transporte seguro y reporting.

DNSSEC se incorporo como una capa adicional de confianza para reforzar la autenticidad e integridad de las respuestas DNS asociadas al dominio. Esto es especialmente relevante porque SPF, DKIM, DMARC, MTA-STS y TLS-RPT se apoyan en DNS para publicar sus politicas o claves.

  • DNSSEC: capa de seguridad para proteger la autenticidad e integridad de las respuestas DNS del dominio.
  • SPF: registro TXT utilizado para declarar que servidores estan autorizados a enviar correo en nombre del dominio.
  • DKIM: publicacion de clave publica asociada a un selector para validar firmas criptograficas de mensajes.
  • DMARC: politica de dominio para indicar que hacer cuando SPF/DKIM no superan las comprobaciones o no estan alineados.
  • MTA-STS: politica para indicar a emisores compatibles que deben entregar correo mediante TLS valido.
  • TLS-RPT: mecanismo de reporting para recibir informacion sobre problemas de transporte TLS.

Esta capa DNS permite que la seguridad de correo no dependa unicamente de la configuracion local del servidor, sino tambien de politicas publicadas y verificables desde el exterior. Por eso el proyecto documento tanto la configuracion como la validacion de estos registros.

SPF, DKIM y DMARC

La autenticacion de correo fue uno de los bloques principales del proyecto. Se trabajo con SPF, DKIM y DMARC para validar si los mensajes estaban autorizados, firmados correctamente y alineados con el dominio visible del remitente.

El objetivo no fue solo publicar registros DNS, sino comprobar su comportamiento mediante pruebas reales y negativas. Por eso el proyecto incluye eventos como SPF fail, DKIM broken, DKIM absent, DKIM rotation y spoofing_reject.

  • SPF: se utilizo para declarar que origenes estaban autorizados a enviar correo en nombre del dominio. Tambien se validaron escenarios de SPF fail.
  • DKIM: se utilizo para firmar mensajes y permitir validacion criptografica mediante selector y clave publica publicada en DNS.
  • Rotacion DKIM: se documento el cambio controlado de selector hacia s2026, manteniendo evidencias de validacion.
  • DKIM broken: se probaron escenarios donde la firma DKIM no era valida, permitiendo comprobar la deteccion de alteraciones o fallos de firma.
  • DKIM absent: se contemplaron escenarios donde el mensaje no incluia firma DKIM, diferenciandolos de firmas rotas.
  • DMARC: se aplico una politica progresiva hasta llegar a p=reject, con el objetivo de rechazar mensajes que no superaran autenticacion y alineacion de dominio.
  • Spoofing reject: se realizaron pruebas externas de suplantacion que terminaron rechazadas por politica DMARC.

Esta parte del proyecto conecta configuracion DNS, comportamiento SMTP, validacion por milters y evidencias SOC. Los resultados no se quedaron solo en configuracion teorica: se tradujeron en registros, pruebas documentadas y metricas visibles en el dashboard.

Postfix, OpenDKIM, OpenDMARC y OpenARC

La capa de servidor de correo se apoyo principalmente en Postfix como MTA. Sobre Postfix se integraron milters defensivos para firmar, validar, rechazar y registrar resultados de autenticacion de correo. Esta integracion fue clave para que las pruebas no se quedasen en teoria, sino que generasen evidencias reales en syslog.

El objetivo de esta parte fue construir un flujo SMTP defensivo donde cada mensaje pudiera ser evaluado por componentes especializados: DKIM, DMARC y ARC. Los eventos generados por estos componentes sirvieron despues como base para el dataset SOC y el dashboard.

  • Postfix: actuo como servidor SMTP principal. Gestiono sesiones SMTP, entregas, rechazos, pruebas de relay, destinatarios inexistentes y eventos de autenticacion.
  • OpenDKIM: se utilizo para firma y validacion DKIM. Permitio trabajar con escenarios de DKIM correcto, DKIM roto, DKIM ausente y rotacion de selector.
  • OpenDMARC: se utilizo para evaluar la politica DMARC del dominio y aplicar rechazo cuando los mensajes no cumplian autenticacion y alineacion. Esta pieza fue clave para las pruebas de spoofing rechazado.
  • OpenARC: se incorporo como parte del ecosistema de autenticacion para contemplar escenarios donde un mensaje pasa por intermediarios, forwarding o servicios que pueden alterar resultados SPF/DKIM.
  • Milters: Postfix se integro con estos filtros para evaluar mensajes durante el flujo SMTP y registrar resultados tecnicos reutilizables como evidencias SOC.
  • Syslog: se utilizo como fuente principal de evidencias para capturar rechazos, fallos, validaciones y resultados de las pruebas.

Esta capa permitio conectar la configuracion tecnica del servidor con evidencias reales. Por ejemplo, un rechazo DMARC no se documento simplemente como una conclusion, sino como evento observable en logs, clasificado despues dentro del dataset SOC como spoofing_reject.

Transporte seguro: STARTTLS, IMAPS, MTA-STS y TLS-RPT

Ademas de autenticar el correo, el proyecto valido la seguridad del transporte. El objetivo fue comprobar que las conexiones SMTP e IMAPS podian negociarse de forma segura y que el dominio publicaba politicas relacionadas con transporte TLS.

Esta parte es importante porque SMTP historicamente permite conexiones oportunistas: si no se valida correctamente, puede haber riesgo de degradacion de seguridad o de entregas sin las garantias esperadas. Por eso se documentaron pruebas STARTTLS, validaciones IMAPS TLS, MTA-STS y TLS-RPT.

  • STARTTLS SMTP: se realizaron pruebas de negociacion TLS sobre SMTP para comprobar que el servidor anunciaba y aceptaba transporte seguro cuando correspondia.
  • IMAPS TLS: se validaron conexiones seguras contra Dovecot para comprobar el canal cifrado en servicios IMAP seguros.
  • MTA-STS enforce: se configuro una politica orientada a exigir TLS valido a emisores compatibles cuando entregan correo al dominio.
  • TLS-RPT: se documento como mecanismo para recibir reportes sobre problemas de entrega TLS, aunque en el proyecto no se dispuso de reportes JSON parseables.
  • Evidencias SOC: las validaciones STARTTLS SMTP e IMAPS TLS se incorporaron como eventos SOC cuando existia evidencia tecnica verificable.

En el dashboard, estas pruebas aparecen en categorias como tls_starttls_smtp y tls_imaps, permitiendo separar los controles de autenticacion de correo de los controles de transporte seguro.

Automatizacion, syslog y dashboard SOC

La capa SOC del proyecto convierte eventos tecnicos del servidor en informacion estructurada, revisable y visual. El objetivo fue que la configuracion defensiva no quedara solo como una instalacion tecnica, sino como un flujo completo de deteccion, procesamiento, evidencia y visualizacion.

La fuente principal de evidencias fue syslog. Desde ahi se extrajeron eventos generados por Postfix, OpenDMARC, OpenDKIM, Dovecot, autenticacion SMTP, rechazos, pruebas TLS, DKIM, SPF, DMARC y escenarios de abuso SMTP.

  • Syslog: fuente principal de eventos reales del servidor.
  • Procesamiento automatizado: scripts propios para detectar eventos relevantes y transformarlos en registros estructurados.
  • Dataset SOC: CSV con pruebas, categorias, resultados, ubicacion final y observaciones sanitizadas.
  • Criterio de inclusion: una prueba solo se registraba si existia evidencia tecnica verificable.
  • Dashboard Python/Dash: visualizacion de KPIs, tablas, categorias, evidencias y resultados defensivos.
  • Version publica: datos sanitizados para portfolio, sin exponer IPs sensibles, credenciales ni rutas internas.

Este flujo permite explicar el proyecto como una cadena defensiva completa: configuracion del servidor, generacion de logs, clasificacion SOC, normalizacion de datos, metricas y visualizacion final.

En terminos de portfolio, esta parte demuestra capacidades practicas de Blue Team: analisis de logs, tratamiento de evidencias, automatizacion, scripting, modelado de eventos y comunicacion visual de resultados mediante dashboard.

Pruebas SOC

Las pruebas SOC fueron disenadas para validar el comportamiento defensivo del servidor de correo ante situaciones reales o controladas. La idea no fue generar volumen artificial, sino registrar eventos que pudieran justificarse mediante evidencias tecnicas.

Cada prueba incorporada al dataset debia contar con al menos una evidencia verificable: respuesta SMTP, registro syslog, transcript TLS, consulta DNS, validacion de milter, rechazo de Postfix/OpenDMARC/OpenDKIM o resultado tecnico reproducible.

  • auth_fail: intentos de autenticacion SMTP fallida registrados desde Postfix/SASL.
  • recipient_unknown: pruebas contra destinatarios inexistentes y rechazos SMTP asociados.
  • open_relay_test: validacion de que el servidor no actuaba como relay abierto.
  • tls_starttls_smtp: validaciones STARTTLS SMTP para comprobar transporte seguro.
  • tls_imaps: validaciones TLS sobre IMAPS/Dovecot.
  • spf_fail: escenarios de origen no autorizado segun SPF.
  • dkim_broken: pruebas con firma DKIM alterada o invalida.
  • dkim_absent: escenarios sin firma DKIM.
  • dkim_rotation: validacion de cambio de selector DKIM.
  • lookalike_domain: analisis de dominios visualmente similares al dominio legitimo.
  • spoofing_reject: intentos externos rechazados por politica DMARC.

Estas pruebas permitieron transformar actividad tecnica del servidor en eventos SOC categorizados. El resultado final fue un dataset con 190 eventos SOC procesados, que alimenta el dashboard y permite explicar la postura defensiva del laboratorio.

Spoofing y DMARC

Una parte clave del proyecto fue comprobar la proteccion del dominio frente a intentos de suplantacion. Para ello se aplico una politica DMARC progresiva hasta llegar a p=reject, validando que el servidor rechazaba mensajes que no superaban autenticacion y alineacion.

Las pruebas externas de spoofing no buscaban entregar mensajes, sino comprobar que Postfix y OpenDMARC aplicaban correctamente la politica del dominio. Los rechazos se registraron como evidencias tecnicas en syslog y fueron incorporados al dataset SOC bajo la categoria spoofing_reject.

  • DMARC p=reject: politica final aplicada para rechazar mensajes no alineados o no autenticados.

  • Postfix: servidor encargado de aplicar el rechazo final y registrar el evento.

Esta validacion es importante porque demuestra que el dominio no solo tenia una politica DMARC publicada, sino que dicha politica fue probada con intentos de suplantacion y produjo rechazos observables.

Privacidad

Para publicar el proyecto como portfolio se preparo una version publica sanitizada. El objetivo fue conservar el valor tecnico del trabajo sin exponer informacion sensible del servidor original, del dominio, de usuarios, IPs o configuraciones privadas.

La version publica mantiene arquitectura, metodologia, categorias de pruebas, metricas, documentacion y dataset SOC anonimizados, pero elimina cualquier elemento que pudiera comprometer la seguridad del entorno real.

  • IPs anonimizadas o sustituidas por marcadores genericos.
  • Rutas internas del servidor eliminadas o redactadas.
  • Correos externos anonimizados.
  • Sin claves privadas DKIM.
  • Sin contrasenas ni credenciales.
  • Sin ficheros .env.
  • Sin tokens ni sesiones sensibles.
  • Sin backups completos del servidor.
  • Dataset SOC publico con observaciones sanitizadas.

Este enfoque permite mostrar capacidades reales de Blue Team, SOC, email security, documentacion y automatizacion, manteniendo una separacion clara entre evidencias publicables y material sensible que debe permanecer fuera del repositorio.
